# Crypto Quant 项目设计讨论

## 1) 产品目标

本项目的目标不是只做一个“能跑起来”的脚本，而是搭建一个后续可以逐步演进为研究、回测、模拟盘、实盘一体化的加密量化框架。第一阶段聚焦 BTC/ETH 等主流现货或永续合约市场，优先支持 Binance 与 OKX 两家交易所，覆盖基础行情接入、策略开发、回测验证、风控约束、人工操作与命令行工作流。

从产品视角看，需要同时服务三类使用场景。第一类是研究型场景，要求能够快速迭代策略假设、读取历史数据、验证收益曲线和风险指标；第二类是执行型场景，要求统一的交易所适配层、清晰的订单与持仓抽象，以及可追踪的日志；第三类是运营型场景，要求手工介入、配置管理、参数切换、异常处理和回放分析都有落点。因为项目还处于起步阶段，首版必须保持骨架清晰、依赖克制、模块边界稳定。

技术选型上，首批推荐使用 `ccxt` 统一交易所接口，使用 `pandas` 处理 OHLCV 与回测数据，使用 `pydantic` 管理配置和参数校验，使用 `typer` 搭建 CLI 工作流，使用 `pytest` 建立最小测试面。等研究与执行复杂度提升后，可以再考虑把 `vectorbt`、`backtrader`、`freqtrade` 作为未来的增强方向，而不是一开始就将系统绑定到大型框架。

## 2) 系统架构

系统建议分为五层。第一层是配置层，负责环境参数、交易所凭证、策略参数、风控阈值和回测配置的解析与校验；第二层是适配层，封装 Binance/OKX 这类外部交易所的行情与交易接口；第三层是策略层，负责将市场数据映射成标准化信号；第四层是执行与风控层，负责仓位控制、止盈止损、最大回撤约束、订单下发和失败处理；第五层是接口层，首版以 CLI 为主，后续再扩展到 Web 面板、任务调度和告警通知。

为了降低后续重构成本，策略模块不应直接依赖 `ccxt`，而应依赖统一的 `ExchangeAdapter` 接口和标准化的数据结构。这样做有两个好处：一是研究与实盘可以共享策略逻辑，二是未来如果要切换到 WebSocket、自建撮合回放、甚至接入第三方执行平台，改动主要落在适配层而不是策略层。

初版不必过早引入消息队列或微服务，但模块边界应该允许以后拆分。例如今天 CLI 直接串联“读配置 -> 取数据 -> 生成信号 -> 回测/下单”，以后可以平滑替换成调度器驱动的事件流。也就是说，首版可以是单进程，但代码组织要为多进程或异步架构留接口。

## 3) 策略模块，包括套利与非套利量化策略

策略层至少分成两大类。第一类是非套利策略，典型包括趋势跟踪、均值回归、突破、动量轮动等。首版可以先用简单均线交叉策略作为演示，因为它结构清晰、数据要求低，适合验证配置、信号、仓位和回测链路是否闭环。即使它不一定具备真实可交易性，也足够作为工程基线。

第二类是套利策略，例如跨交易所现货价差套利、现货与永续基差套利、资金费率套利、三角套利等。套利策略通常比非套利策略更依赖执行细节，因为收益空间往往被手续费、滑点、提现延迟、挂撤单失败和仓位不平衡迅速吞噬。首版适合先实现“跨 Binance/OKX 价差监测与信号占位”的框架，不急于宣称可实盘盈利。也就是说，策略模块需要能表达“进入套利”“退出套利”“等待机会”这类状态，但执行层仍以占位逻辑为主。

为了让策略开发具有统一接口，建议每个策略都遵循相同约束：接收标准 `DataFrame` 数据，输出标准 `Signal` 对象，参数通过 `pydantic` 校验，避免散落的字典访问。策略内部只关心业务逻辑，不直接访问环境变量或命令行参数。未来若引入 `vectorbt` 或 `backtrader` 做高性能研究，也可以让这些框架作为回测后端，而不是让策略逻辑被框架 API 绑死。

## 4) OKX / Binance 交易所 API 集成计划

交易所集成优先采用 `ccxt`，因为它对 Binance 与 OKX 已提供相对统一的 REST 访问方式，适合快速建立公共行情访问、账户查询和基础下单骨架。需要明确的是，统一接口并不代表细节完全一致。OKX 常见地需要 `password/passphrase`，Binance 与 OKX 的测试环境、限频策略、符号命名、某些订单参数和私有接口权限都会有差异，因此适配层仍然要保留交易所特定配置。

集成计划可以分三步。第一步，只接入公开行情接口，例如 `fetch_ticker` 和 `fetch_ohlcv`，用于策略研究与手工联调。第二步，增加私有 API 的最小封装，包括账户余额读取和下单接口，但默认不启用自动交易，而是通过 CLI 手工触发。第三步，再引入更稳健的重试、超时、签名失败告警、订单状态轮询与审计日志。这样能避免项目在一开始就被私有交易细节拖住。

同时要考虑未来的升级路径。REST 适合原型阶段，但当策略依赖盘口、逐笔成交、低延迟撤单时，后面应该逐步评估 WebSocket、专用撮合事件流和本地缓存层。首版架构里只需要保证适配器接口可扩展，例如今天实现 `fetch_ohlcv`，以后可以无缝加入 `subscribe_order_book` 或 `stream_trades`。

## 5) 回测与收益验证

回测模块的目标不应只是算出一个收益率，而是验证策略是否满足交易成本、资金约束和执行可达性。首版建议先支持基于 `pandas` 的轻量回测器，输入为 OHLCV 或价差序列，输出资金曲线、交易次数、胜率、总收益、最大回撤等关键指标。由于项目初期重点是工程闭环，回测器可以先保持简化，例如只处理单品种、单策略、固定手续费和固定滑点。

收益验证必须明确区分“理论信号收益”和“可执行收益”。尤其是套利策略，即使历史价差看起来明显，也需要扣掉双边手续费、买卖盘冲击、转账延迟和仓位对冲失败的风险。换句话说，回测报告里至少要保留手续费、滑点和持仓变化的可解释记录，而不是仅输出净值曲线。

等系统成熟后，可以评估接入 `vectorbt` 提升参数扫描效率，或接入 `backtrader` / `freqtrade` 获得更成熟的回测与实盘桥接能力。但在第一阶段，不建议把这类框架作为唯一核心，因为当前更重要的是先掌握策略逻辑、适配器边界和风控链路，而不是过早增加框架复杂度。

## 6) 止盈 / 止损 / 风险控制

风险控制应独立于策略存在。策略负责发出“想做什么”的信号，风控负责判断“允许做多少、是否还能继续做、什么时候必须退出”。首版至少需要四类约束：单笔名义仓位上限、最大同时持仓数、固定比例止损、固定比例止盈。即使只是回测，也要把这些规则纳入统一的 `RiskManager`，否则后续实盘容易出现研究与执行行为不一致的问题。

对于非套利策略，可以先用相对简单的价格型风控。例如开仓后若回撤超过 `stop_loss_pct` 则强制平仓，若收益达到 `take_profit_pct` 则分批或一次性止盈。对于套利策略，还需要额外关注腿不对称、单边成交、交易所可用余额、提币冻结、费率变化和 API 延迟等复合风险。首版虽然可以先做占位逻辑，但文档和接口要预留这些扩展点。

此外，风控不只是一组阈值，还需要可解释输出。每次拒单、降仓、止损触发、达到日内亏损阈值，都应记录原因。后续如果接入告警系统，这些风控事件会是最重要的消息源之一。

## 7) 运维 / 人工操作 / 用户工作流

项目首版建议以 CLI 为中心，因为它最适合研究和半自动流程。基础工作流可以是：编辑配置文件 -> 运行 `show-config` 验证配置 -> 运行 `backtest` 校验策略逻辑 -> 如需联调，再手工执行公开行情查询或模拟下单命令。这样做能把“研究确认”和“执行动作”明确区分开，降低误操作概率。

从运维角度看，建议默认采取“人工审核优先”的模式。也就是说，即使未来加入下单能力，也应先支持干跑、模拟盘、手工确认、订单日志落地，再逐步演进到定时任务或守护进程。对加密量化项目来说，交易所权限配置错误、测试网与正式网切换失误、凭证泄漏和错误仓位通常比策略本身更容易造成严重损失。

用户工作流还应尽量标准化。配置文件统一放在 `configs/`，策略参数通过 `pydantic` 校验，命令行输出保持可读、可复制、可记录。测试方面，使用 `pytest` 维护最小的单元测试与冒烟测试，确保配置、策略和回测基础功能不会在后续迭代中静默损坏。

## 8) 分阶段实施路线图

第一阶段是当前要完成的内容：建立工程骨架、配置管理、`ccxt` 适配器、均线策略、套利占位策略、简化回测器、风控骨架、`typer` CLI、示例配置和 `README`。这一步的成功标准不是收益率，而是工程可运行、模块边界清晰、能够支持后续迭代。

第二阶段聚焦研究增强。具体包括：接入真实历史数据源、补充绩效指标、支持多品种与多策略组合、增加参数扫描、补充更多非套利策略和更细的套利观测逻辑。如果届时研究需求明显提升，可以开始评估 `vectorbt` 作为高性能研究引擎。

第三阶段聚焦执行与风控增强。包括：账户余额同步、真实订单生命周期管理、重试与异常审计、WebSocket 行情、订单簿与滑点模型、风控告警、定时调度和运行日志归档。若目标逐步靠近可实盘系统，则可以评估 `backtrader` 或 `freqtrade` 是否在实盘桥接、策略生态或运维工具链方面提供额外价值，但前提仍然是当前自定义骨架已经验证有效。

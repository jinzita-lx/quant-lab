hermes: 这次我们不继续谈泛泛而谈的“量化框架”，只做更深的规划。先定一个基调：以这个仓库当前的状态，所谓“套利策略”到底应该被理解成什么？

codex: 以当前仓库的实现边界来看，`spread_arbitrage` 还只是“价差信号占位策略”，不是可实盘套利系统。仓库里现在有 `ccxt` 公共 REST 行情、`quote` 命令、简化回测器、基础 `RiskManager` 和一个只根据两个价格列判断 `enter_spread` / `exit_spread` 的策略骨架，但没有订单状态机、没有双腿持仓模型、没有私有 API 对账、没有 WebSocket、没有 paper trading，也没有真实的撮合和成交回放。所以文档和后续实现都应该非常克制：第一阶段把套利理解为“机会识别、成本建模、执行可行性验证”，而不是“看到价差就能赚钱”。如果这个基调不立住，后面所有回测结果都会高估。

hermes: 那你把套利策略分类讲透。跨交易所现货价差、现货和永续基差、资金费率 carry、三角套利，这四类分别是什么，真正的难点又卡在哪里？

codex: 第一类是跨交易所现货价差，也就是 Binance 和 OKX 同一现货交易对在同一时间的报价不同，理论上可以在便宜的地方买、贵的地方卖。它的优点是概念最直观，和当前仓库已有的 `spread_arbitrage` 占位策略最接近；缺点是实盘几乎不可能靠“买完再转币过去”完成闭环，真正可做的前提是两边都预先放好库存和保证金，然后靠库存再平衡维持可交易状态，所以它本质上是库存管理问题，不是单纯信号问题。第二类是现货和永续基差，常见形式是买现货、卖永续，赚基差收敛或配合资金费率；它比跨所现货更接近真实可执行，因为不依赖链上转账，但它要求你接好永续合约 API、杠杆模式、保证金规则和强平风险，工程复杂度明显更高。第三类是资金费率 carry，本质上是基差交易的一个收益来源视角：当某一侧永续合约长期支付正 funding 时，你持有相反方向的永续头寸并配对现货或另一腿对冲，赚 funding 净流入；这类策略看起来“稳”，但真正的坑在于 funding 并不稳定，basis 可能扩大，借币成本、手续费和滑点会把纸面收益吃掉。第四类是三角套利，例如 `BTC/USDT`、`ETH/USDT`、`ETH/BTC` 三个市场之间形成乘法不一致，这类策略最依赖盘口深度、撮合延迟、挂撤单速度和手续费等级，在头部交易所上通常已经被低延迟团队和做市系统吃得很薄了，用当前这种 Python CLI 骨架直接上，基本不现实。

hermes: 如果只允许选一个“最现实的首版”，你会落哪一个？我要的不是教材答案，而是对这个仓库最有价值的答案。

codex: 如果问题是“当前仓库下一步先把什么做出来最有价值”，我的答案是先做跨交易所现货价差的监控、回测和 paper execution，而不是直接做 live arbitrage。理由很简单：仓库已经有 `spread_arbitrage` 策略骨架，现货 REST 行情最容易打通，回测输入也只需要双边价格序列，这能最快把数据模型、成本模型、双腿订单抽象和风控事件补齐。可如果问题变成“哪一种套利更接近未来可交易”，那我会把长期优先级放到同一交易所内的现货-永续基差或 funding carry，因为它不依赖跨所转账和库存搬运，落地时更像交易系统问题而不是清算系统问题。三角套利我会明确放弃第一阶段，甚至第二阶段也不碰；市场做得太干净，工程投入和收益预期完全不成比例。

hermes: 套利之外，这个项目在 SMA 之后最值得加的非套利策略是什么？我不想把仓库变成策略动物园，所以你要有取舍。

codex: 先加突破，再加均值回归，然后才考虑动量，做市放到很后面。突破策略最适合当前仓库，因为它只依赖 OHLCV，和现有 `backtest` 的 K 线级假设兼容，能直接检验“入场、出场、止损、滑点、持仓限制”这条链路是否靠谱；它也比均线交叉更接近真实交易决策，因为你可以围绕区间高低点、ATR、成交量确认来构建更明确的执行条件。均值回归值得第二个做，因为它能逼着回测器处理更短持有周期、更高换手和更严格的成本敏感性，这对整个研究层是好事。动量适合在你补齐多品种历史数据之后再做，特别是做横截面动量时，单品种回测器就不够用了。至于市场做市，我不建议在近期路线里排高优先级；它需要订单簿、排队位置、maker/taker 费率、撤单节奏、库存倾斜、毒性流量识别和异常保护，远不是在 K 线级别上写一个“挂 bid/ask”逻辑就算完成。对这个仓库而言，做市最大的价值不是尽快实现，而是提醒你哪些基础设施现在还完全没有。

hermes: 你把 Binance 和 OKX 的 API 集成讲细一点。REST 和 WebSocket 怎么分工，认证差异、限频、测试网、时钟、符号、订单生命周期这些到底应该怎么设计？

codex: 分工上我建议非常明确：REST 负责历史数据抓取、交易所元数据、一次性查询、补单查询和低频控制动作，WebSocket 负责实时行情、订单簿增量、私有订单回报和账户变更。不要让 REST 承担持续轮询盘口的职责，也不要把 WebSocket 当成唯一真相源；实盘一定是“WebSocket 增量流 + REST 周期校验”的双轨模型。认证方面，Binance 和 OKX 在 `ccxt` 之上虽然能被包装成相似接口，但差异不能假装不存在。Binance 更典型的是基于毫秒时间戳和签名窗口，时钟漂移会直接触发请求被拒；OKX 则除了 key/secret 还多一个 passphrase，私有接口、demo 盘标识和某些产品线参数也更容易踩坑。限频方面，`enableRateLimit` 只能算保底，远远不够，真正要做的是按“公共行情、私有查询、下单、撤单”分别建预算桶，并把突发重试、批量任务和多 symbol 订阅算进去。测试环境也要谨慎看待：Binance 的 testnet 和 OKX 的 demo 主要适合验证签名、权限和下单链路，不适合验证策略利润，因为流动性、撮合行为和延迟分布都和真实环境差很多。

hermes: 把工程细节再掰开一点。我尤其关心时钟同步、符号归一、下单重试、幂等和错误处理，不想将来靠“人肉猜”订单状态。

codex: 时钟同步必须做成守门人机制，而不是一个可选优化。启动时先取交易所服务器时间，记录本地偏移；运行中定期刷新，如果偏移超过你设定的阈值，比如 300 到 500 毫秒，就拒绝新的私有请求并触发告警。符号归一不能停留在字符串层面，内部应该有统一的 `InstrumentId`，至少包含 `venue`、`market_type`、`base`、`quote`、`settle`、`contract_type`、`tick_size`、`lot_size` 和 `min_notional` 这些字段；否则 `BTC/USDT`、`BTC-USDT`、`BTC-USDT-SWAP`、`BTCUSDT` 混在一起后，回测和执行迟早会对不齐。订单生命周期建议从第一天起就显式建模成 `created -> submitted -> acknowledged -> partially_filled -> filled / canceled / rejected / expired`，再加一个 `unknown` 兜底状态用于网络中断后的恢复。重试策略要区分动作类型：`GET` 类查询和部分可重入的撤单可以安全重试；`POST create_order` 在超时后不能盲目重发，否则最容易制造重复订单。这里的关键是客户端订单号，也就是你自己生成稳定的 `client_order_id`，让“提交请求超时但交易所可能已接收”的情况能通过查询这个 id 做对账。错误处理也必须分类：认证错误、参数错误、限频、余额不足、最小下单量不满足、只减仓冲突、post-only 被吃单、网络抖动、交易所不可用、服务端内部错误，这些都不能用一个 `except Exception` 吞掉。文档里要明确写清楚每一类错误的动作策略，是重试、降级、告警、人工接管还是直接 kill switch。

hermes: 收益验证这块你怎么设计方法论？我不想看到“回测收益 80%”这种没有成本和时延意识的东西。

codex: 收益验证必须分成三层：研究层、模拟执行层、准实盘层。研究层的最低要求是把回测假设写死在报告里，例如信号在 K 线收盘时产生、在下一根 K 线开盘成交，还是在本根收盘按某个滑点成交；如果这是套利策略，成交价不能用 mid 或 `last` 糊弄过去，应该至少区分 bid/ask 或者对双腿分别施加不同的冲击成本。成本模型不能只写一个全局 `fee_rate`，最终至少要拆成 maker/taker 手续费、资金费率、借币或融资成本、跨所提现和再平衡成本，以及极端行情下的额外冲击。滑点也不该永远固定成一个常数，第一版可以保留固定 bps，但必须做敏感性分析，比如 2bps、5bps、10bps、20bps 各跑一遍，看策略是否一碰成本就死。延迟方面，要至少建两个参数：信号产生到下单的延迟、下单到成交的延迟，并让价差策略能在延迟窗口里重新评估机会是否已经消失。

hermes: 继续，把那些最容易自欺欺人的陷阱说透，尤其是 lookahead、幸存者偏差、数据泄漏、walk-forward 和 paper trading。

codex: 最常见的错误是把未来信息偷偷喂进策略里。比如用整根 K 线的 `high`、`low` 或最终收盘价去决定同一根 K 线内的成交，这就是典型 lookahead；又比如 funding-rate carry 用的是已经结算出来的 funding，而不是交易当时可观察到的下一期预估 funding，也属于数据泄漏。幸存者偏差在加密市场尤其严重，因为很多小币种后来消失了，回头只拿今天还活着的交易对做筛选，结果会明显乐观。跨交易所数据还要警惕时间戳对齐问题，Binance 和 OKX 的同一分钟 K 线不一定严格同时闭合，拿错时间基准会人为制造“可套利”的假象。我的建议是把验证流程固定成：先做样本内研发，再做滚动 walk-forward，把参数冻结后推进到下一段样本外，最后进入 paper trading。paper trading 的目标不是再看一遍净值，而是比较“理论成交”和“模拟撮合成交”的偏差，观察滑点漂移、拒单率、部分成交率、下单延迟分布和机会衰减速度。指标上，除了净收益，至少要统一输出年化 Sharpe、最大回撤、胜率、换手率、容量估计和持仓时间；其中容量不是可选项，特别是套利和高换手策略，如果订单规模稍微放大就明显推高冲击成本，那它就不是一个可以放大资金的策略。

hermes: 风控部分不要只讲止盈止损。把最大敞口、熔断、日内亏损上限、kill switch、部分成交、腿风险和资金分配都纳进来，给我一个能操作的框架。

codex: 我建议把风控拆成四层。第一层是订单前风控，检查单笔名义价值、账户可用余额、合约杠杆限制、最小下单单位、单策略剩余额度和组合层总敞口；这一层应该在信号刚产生时就拦住明显不该发出去的单。第二层是持仓中风控，针对趋势或均值回归策略用价格止损、追踪止盈和时间止损，针对套利策略则优先看价差回归失效、basis 扩张、funding 反转和对冲腿失配，而不是只看单边价格涨跌。第三层是系统级风控，包括日内亏损上限、连续拒单阈值、数据陈旧检测、时钟漂移超阈值、WebSocket 断流、交易所状态异常和人工 kill switch；只要命中这些条件，就应该暂停新开仓，必要时触发主动减仓。第四层是组合和资金分配，明确每个策略只能占用多少资本、允许多少相关性暴露、在波动飙升时是否自动降杠杆。部分成交和腿风险要单独写规则：一条腿成交、另一条腿失败时，系统不能假装这还是“套利仓位”，而要立即进入应急模式，例如用更宽容的 IOC/市价对冲残腿，或者直接按预设损失限额止损。对当前仓库来说，`RiskManager` 现在只处理单腿入场和简单止盈止损，下一步必须升级成能理解“策略类型、组合暴露、订单状态和系统健康度”的组件，否则所谓风控只是参数表。

hermes: 给我一个面向操作者的实际工作流，从配置开始，一直到 quote test、backtest、dry-run，再到未来 live trading。注意区分哪些是这个仓库现在已经有的，哪些是你建议补出来的。

codex: 我会把操作者手册设计成五个阶段。第一阶段是配置准备，现在就能做：创建虚拟环境、安装依赖、复制 `configs/example.toml` 为本地配置、设置环境变量形式的 API 凭证，然后用 `quant-lab show-config --config configs/your.toml` 检查脱敏后的配置是否符合预期。第二阶段是行情连通性验证，现在也能做：分别执行 `quant-lab quote --config configs/your.toml --exchange binance --symbol BTC/USDT` 和 `quant-lab quote --config configs/your.toml --exchange okx --symbol BTC/USDT`，确认公共行情可以返回、符号写法没有问题、测试网配置不会把公共接口搞坏。第三阶段是研究与回测，现在能做但能力有限：准备符合 schema 的 CSV，针对 `moving_average` 或 `spread_arbitrage` 跑 `quant-lab backtest`，然后手工记录不同手续费和滑点参数下的敏感性结果。第四阶段是 dry-run，这一步当前仓库还没有，需要补 `paper trading` 或 `dry-run` 子命令；我建议它先只做“读实时行情 -> 生成信号 -> 记录模拟订单与风控事件 -> 不发真实订单”，并把所有决策写入日志或数据库。第五阶段才是未来 live trading，而 live 之前的硬门槛应该明确写出来：私有 REST 能稳定查询账户和订单、客户端订单号幂等生效、订单状态机与对账完成、kill switch 生效、paper trading 至少稳定跑过一段连续时间，并且 paper 和理论回测的偏差在可接受范围内。没有这几个条件，就不应该把 `create_order` 接到自动化调度上。

hermes: 最后回到当前仓库，你给一个明确的实现优先级列表。不要空泛，也不要把 Web 面板之类的外围工作排到前面。

codex: 第一优先级是补数据与交易对象模型，也就是统一 `Instrument` / symbol normalization、区分现货与永续、补交易所元数据加载、明确 CSV schema 和时间对齐规则；没有这层，后面无论套利还是非套利都会反复返工。第二优先级是升级回测器，让它支持双腿套利持仓、更多绩效指标、成本分项、延迟假设和 walk-forward，而不是继续在一个单腿 `cash + quantity` 模型上叠功能。第三优先级是实现 paper trading 和订单状态机，把策略输出、风控决策、模拟订单、部分成交和对账事件串起来；这一步是从研究系统走向执行系统的真正分水岭。第四优先级才是 Binance 和 OKX 的私有 REST 集成，包括账户查询、下单、撤单、客户端订单号、错误分类和审计日志。第五优先级是实时层，也就是 WebSocket 行情和私有流，再加上数据陈旧检测、断流重连和 REST 校验。策略路线方面，我建议在非套利里先补 breakout，再补 mean reversion；套利里先把跨所现货价差做成严肃的监控和 paper execution，再决定是否进入同所现货-永续 basis 或 funding carry。市场做市、Web UI、复杂调度系统都应该后移，因为它们现在不会比“更可靠的研究和模拟执行”更值钱。
